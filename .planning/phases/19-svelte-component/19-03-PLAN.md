```svelte
<script>

  import {
    createExtensionSigner,

  import ToolsResources from './components/ToolsResources.svelte';
  import DeployerWidget from '@nsite/deployer/components/DeployerWidget.svelte';

  // SPA-level session store for Navbar display
  // This is separate from DeployerWidget's internal stores
  const spaStores = createDeployerStores({ storagePrefix: 'nsite' });
  const { session } = spaStores;

  let showLoginModal = false;
  let currentSigner = null;
  let deployNsec = null;

  // When DeployerWidget reports auth changes, update SPA session for Navbar
  function handleAuthChange(e) {
    const { pubkey, signerType, authenticated } = e.detail;
    if (authenticated) {
      session.update(s => ({ ...s, pubkey, signerType }));
    } else {
      session.set({ pubkey: null, signerType: null, displayName: null, avatar: null, npub: null });
      currentSigner = null;
      deployNsec = null;
    }
  }

  function handleDeploySuccess(e) {
    // SPA can react to deploy success — e.g., show celebration
    console.log('Deploy success:', e.detail);
  }
</script>
```

**IMPORTANT NOTE on store sharing:** The SPA's App.svelte and DeployerWidget will share the same localStorage keys (both use prefix 'nsite'). This means session state persisted by the widget is visible to the SPA's Navbar on next load. The `session` store in App.svelte is created via the same `createDeployerStores` factory — this is intentional to keep the Navbar in sync.

However, the SPA does NOT pass its session store into DeployerWidget. Instead, DeployerWidget creates its OWN stores internally. Both read from the same localStorage, so they stay in sync. The `signer` prop is the only runtime bridge.

**Alternative approach (simpler, recommended):** Instead of creating a separate store instance in App.svelte, import the DeployerWidget without a signer prop and let it manage its own auth entirely. The Navbar can be simplified to not show session info, or can read from the same localStorage directly.

For the recommended approach, App.svelte becomes even simpler:

```svelte
<script>
  import Navbar from './components/Navbar.svelte';
  import NIP5ABanner from './components/NIP5ABanner.svelte';
  import ToolsResources from './components/ToolsResources.svelte';
  import { DeployerWidget } from '@nsite/deployer';

  function handleDeploySuccess(e) {
    console.log('Deploy success:', e.detail);
  }
</script>
```

Choose the approach that keeps Navbar functional. Read `Navbar.svelte` to determine what session data it needs and whether it can be supplied by DeployerWidget's auth-change events or if it needs its own store.

**Key decisions based on Navbar inspection:**
- If Navbar imports `session` from store.js directly, it needs refactoring to accept session data as props
- If Navbar only needs pubkey/displayName/avatar, these can be passed as props from App.svelte which listens to DeployerWidget's auth-change events

**Refactor Navbar.svelte** to accept session data as props instead of importing the store:


Then App.svelte passes these from its local state, updated by DeployerWidget's auth-change events.
