<action>
Update each component's imports from relative paths to @nsite/deployer sub-path imports. For components that import `session`, `deployState`, or `serverConfig` (the old singletons), they must instead receive them as props from App.svelte, OR use `createDeployerStores()` locally. Given that App.svelte already creates the stores, the cleanest approach for Phase 18 is: **components that need stores should import `createDeployerStores` and create their own instance with the same default prefix**, OR App.svelte should pass stores as props.

**IMPORTANT DECISION:** For Phase 18, the simplest correct approach is to have each component that uses stores call `createDeployerStores()` with the same default prefix. Since `persistedStore` reads from localStorage on creation, all instances reading `nsite_session` will get the same initial value. Updates propagate through localStorage but NOT through Svelte reactivity across instances. This is acceptable for Phase 18 because:
- The SPA currently has only ONE instance of each component on-screen at a time
- Phase 19 will introduce Svelte context to share a single store set within the DeployerWidget component tree

The alternative (passing stores as props) would require modifying every component's prop interface AND every parent's template — a much larger diff for Phase 18. The factory call per component is functionally identical to the current singleton behavior for single-instance usage.
