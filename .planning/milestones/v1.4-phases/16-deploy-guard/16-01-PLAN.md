# Plan 16-01: Deploy Guard — Site overwrite warnings and loading state block

<objective>
Add inline deploy guards to the SPA's review step that warn users when they're about to overwrite an existing root or named site, offer an "Update" shortcut to enter the update flow, and block the deploy button while site data is still loading. Users with no existing sites experience zero friction.
</objective>

<background>
The deploy flow currently lets users select a site type (root/named) and deploy without any awareness of whether that site already exists. A user could accidentally overwrite their existing root site or a named site by deploying again. The manage view already has an update flow (via `on:update` handler) that sets up siteType, dTag, title, description, and calls `resetForUpdate()`. The deploy guard reuses this exact pattern but triggers it from the deploy review step.

All site data is already available in `allSites` (fetched by `fetchSiteInfo()` on mount/login). The `sitesLoading` boolean already exists but isn't wired into `canDeploy`.
</background>

<action>
In the Derived section of App.svelte (after the `existingFileCount` reactive declaration around line 200), add a new reactive declaration:

```js
// Deploy guard: detect if entered dTag matches an existing named site
$: matchingNamedSite = siteType === 'named' && dTag && dTagValid && !dTagReadOnly
  ? allSites.named.find(s => getManifestDTag(s) === dTag)
  : null;
```

The `!dTagReadOnly` check prevents the warning from showing when the user is already in the update flow (routed from manage view with `dTagReadOnly = true`).

In the Event handlers section (near `handleFilesSelected` around line 213), add a helper function:

```js
function handleGuardUpdate(site) {
  if (site.kind === 35128) {
    siteType = 'named';
    dTag = getManifestDTag(site) || '';
    dTagReadOnly = true;
  } else {
    siteType = 'root';
    dTag = '';
    dTagReadOnly = false;
  }
  deployTitle = getManifestTitle(site);
  deployDescription = getManifestDescription(site);
  resetForUpdate();
}
```

This is the same logic as the manage view's `on:update` handler — extracted to a named function so both the root guard and named guard can call it.
</action>
