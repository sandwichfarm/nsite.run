## Phase Details

<details>
<summary>✅ Phase 12: Local Development Harness — SHIPPED 2026-03-22</summary>

**Goal**: Developers can run the full nsite.run stack locally with a single command, using local storage backends that mirror production routing architecture
**Depends on**: Phase 11
**Requirements**: DEV-01, DEV-02, DEV-03, DEV-04, DEV-05, DEV-06, DEV-07
**Success Criteria** (what must be TRUE):
  1. Developer runs `deno task dev` from the repo root and all four services (relay, blossom, gateway, SPA) start with colored per-service output
  2. Edge scripts (relay, blossom, gateway) execute locally via a `Bunny.v1.serve()` polyfill with no Bunny platform dependency
  3. The local relay stores events in a SQLite file and the local blossom stores blobs in a filesystem directory — both readable on disk after operations
  4. The local gateway routes requests to local relay and blossom instances using the same routing logic as production (WebSocket → relay, blossom endpoints → blossom, subdomain → resolver, root → SPA)
  5. Pressing Ctrl+C terminates all services cleanly with no orphaned processes, and the SPA dev server connects to the local gateway without manual configuration
**Plans**: 3 plans

Plans:

- [x] 12-02-PLAN.md — Gateway dev entrypoint and SPA env configuration
- [x] 12-03-PLAN.md — Root dev orchestrator and deno task dev

</details>

### Phase 13: Leave Confirmation
**Goal**: Users cannot accidentally lose an in-progress deploy or delete operation by closing the tab, reloading, or switching SPA tabs without a warning
**Depends on**: Phase 12
**Requirements**: LEAVE-01, LEAVE-02, LEAVE-03
**Success Criteria** (what must be TRUE):
  1. When a deploy operation is in the hashing, checking, uploading, or publishing step, closing or reloading the browser tab triggers the browser's native leave confirmation dialog
  2. When an in-progress operation is active, clicking the Deploy or Manage tab in the SPA shows an in-page confirmation before switching
  3. When no operation is in progress (idle, reviewing files, on success screen, or on error screen), no leave confirmation appears for any navigation
**Plans**: 2 plans

Plans:
- [x] 13-01-PLAN.md — beforeunload guard and ManageSite delete event dispatch
- [x] 13-02-PLAN.md — OperationBanner component and App.svelte integration

### Phase 14: Delete Animation
**Goal**: Deleting a site gives clear visual feedback at every stage — in-progress, confirmed, failed — with cards animating out cleanly and the list reflowing smoothly
**Depends on**: Phase 13
**Requirements**: DELT-01, DELT-02, DELT-03, DELT-04
**Success Criteria** (what must be TRUE):
  1. After a user initiates delete, the targeted site card becomes visually distinct (e.g., dimmed, "deleting" label) and cannot be expanded or interacted with
  2. Once the relay confirms deletion, the site card fades out and the remaining cards reflow into position without jumping
  3. If deletion fails (all relays reject), the card returns to its normal, fully interactive state — no stuck "deleting" styling
  4. When delete succeeds and the SPA transitions back to the done state, a success or failure outcome is accurately reflected rather than always showing "Deletion complete"
**Plans**: 2 plans

Plans:
- [x] 14-01-PLAN.md — Per-card delete state machine and success/failure distinction
- [x] 14-02-PLAN.md — Card exit animation, smooth reflow, and App.svelte integration

### Phase 15: Post-Action Navigation
**Goal**: After a successful deploy or delete, the user always has an obvious next action available and is never stuck at a dead end with no path forward
**Depends on**: Phase 14
**Requirements**: NAV-01, NAV-02, NAV-03, NAV-04
**Success Criteria** (what must be TRUE):
  1. The Deploy and Manage tab buttons are hidden while a stepper operation is active, preventing disorienting mid-flow navigation
  2. After a successful deploy, a "Manage sites" button is visible on the success screen and navigates to the manage view while preserving the current signer session
  3. After a site is deleted from the manage view, a "Deploy new site" button appears and returns the user to the deploy flow without requiring a page reload
  4. After a non-destructive action completes (e.g., an update), the Deploy and Manage tab buttons reappear so the user can navigate freely
**Plans**: 2 plans

Plans:
- [x] 15-01-PLAN.md — Post-action navigation buttons in SuccessPanel and ManageSite
- [x] 15-02-PLAN.md — Tab button disabling and App.svelte event wiring

### Phase 16: Deploy Guard
**Goal**: Users cannot accidentally overwrite an existing root or named site without a clear warning showing what exists and an explicit confirmation to proceed
**Depends on**: Phase 15
**Requirements**: GUARD-01, GUARD-02, GUARD-03, GUARD-04, GUARD-05
**Success Criteria** (what must be TRUE):
  1. When the user selects the root site option and already has a root site deployed, the option is visually disabled and shows the existing site's URL with an "Update" button
  2. When the user enters a named site dTag that matches one of their existing named sites, an inline warning appears identifying the existing site with an "Update" button
  3. Clicking "Update" in either guard routes the user into the update flow for that specific site (not a fresh deploy)
  4. While site data is still loading after login, the deploy button is blocked with a "checking existing sites..." message — no deploy proceeds on stale data
  5. A user who has no existing sites sees no guards and can deploy freely without friction
**Plans**: 1 plan

Plans:
- [x] 16-01-PLAN.md — Deploy guard warnings and loading state block
