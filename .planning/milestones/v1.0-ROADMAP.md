# Roadmap: nsite.run

## Overview

Six phases take nsite.run from a blank monorepo to a fully operational nsite hosting stack at nsite.run. Phase 1 establishes the build infrastructure before any feature code is written. Phases 2 and 3 port the relay and blossom from working reference implementations. Phase 4 wires up the routing layer and validates cross-script communication on Bunny. Phase 5 implements the nsite resolver and progressive caching state machine — the most complex phase. Phase 6 delivers the Svelte SPA deploy interface. Every phase ships a complete, independently verifiable capability.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 3: Blossom** - nsite-only blossom server live at https://nsite.run blossom endpoints, BUD-01/02 compliant

- [ ] **Phase 6: SPA Deploy Interface** - Svelte SPA at nsite.run root allows authenticated users to deploy an nsite

## Phase Details

### Phase 1: Monorepo and Build Infrastructure
**Goal**: Developers can build all components from a clean checkout and CI catches bundle size violations before any code ships
**Depends on**: Nothing (first phase)
**Requirements**: INFRA-01, INFRA-02, INFRA-03
**Success Criteria** (what must be TRUE):
  1. Running the build command produces separate bundles for relay, blossom, and gateway Edge Scripts
  2. CI pipeline hard-fails when any edge script bundle exceeds 1MB and warns at 750KB
  3. Shared types from `packages/shared` are importable in relay, blossom, and gateway packages with no duplication
  4. A clean checkout produces working builds without manual configuration steps
**Plans**: 3 plans
Plans:
  - [x] 01-01-PLAN.md — Monorepo scaffold, shared package with types/utilities/tests, app stubs
  - [x] 01-02-PLAN.md — esbuild build scripts for all edge scripts, bundle size enforcement script
  - [x] 01-03-PLAN.md — GitHub Actions CI/CD workflows (validate, build, size check, PR delta, Bunny deploy)

### Phase 2: Relay
**Goal**: The nsite-only nostr relay accepts and serves nsite events over standard NIP-01 protocol, and rejects everything outside its scope
**Depends on**: Phase 1
**Requirements**: RELAY-01, RELAY-02, RELAY-03, RELAY-04, RELAY-05, RELAY-06, RELAY-07, RELAY-08
**Success Criteria** (what must be TRUE):
  1. A NIP-01 client can connect to wss://nsite.run, publish kind 15128/35128/10002/10063 events, and query them back
  2. Events of any kind other than 15128, 35128, 10002, 10063 are rejected with a NOTICE
  3. A browser GET to wss://nsite.run with `Accept: application/nostr+json` returns a valid NIP-11 relay info document
  4. nsyte CLI can use wss://nsite.run as a relay target and successfully publish manifest events
  5. Relay events survive a process restart (persisted in Bunny DB)
**Plans**: 2 plans
Plans:
  - [x] 02-01-PLAN.md — Foundation: dependencies, shared constants (kind 5), relay types, DB schema/layer, NIP-11 handler
  - [x] 02-02-PLAN.md — WebSocket relay: rate limiter, NIP-01 handlers (EVENT/REQ/CLOSE), message dispatch, main.ts wiring

### Phase 3: Blossom
**Goal**: The nsite-only blossom server stores and serves blobs for nsite manifests via standard BUD protocol
**Depends on**: Phase 1
**Requirements**: BLSM-01, BLSM-02, BLSM-03, BLSM-04, BLSM-05, BLSM-06, BLSM-07, BLSM-08
**Success Criteria** (what must be TRUE):
  1. A blob can be uploaded via PUT /upload with a valid kind 24242 auth event and retrieved via GET /{sha256}
  2. An upload not referenced by any nsite manifest is accepted (manifest check deferred to v2 per decision)
  3. A blob list for a pubkey is returned via GET /list/{pubkey}
  4. nsyte CLI can use https://nsite.run as a blossom upload target and successfully upload site blobs
  5. HEAD /{sha256} returns 200 for an existing blob and 404 for a missing one
**Plans**: 3 plans
Plans:
  - [ ] 03-00-PLAN.md — Wave 0: test stubs for all 7 modules (auth, storage client, handlers)
  - [ ] 03-01-PLAN.md — Foundation: deno.json imports, types, util, storage client/metadata, auth (nostr-tools), CORS
  - [ ] 03-02-PLAN.md — Handlers (blob-get/upload/delete/list, mirror, upload-check, report, server-info), router, main.ts, build

### Phase 4: Gateway Routing Layer
**Goal**: All traffic arriving at nsite.run is routed to the correct backend by the gateway Edge Script, with no request type leaking to the wrong handler
**Depends on**: Phase 2, Phase 3
**Requirements**: ROUTE-01, ROUTE-02, ROUTE-03, ROUTE-04, ROUTE-05
**Success Criteria** (what must be TRUE):
  1. A WebSocket connection to wss://nsite.run is proxied to the relay and remains open for NIP-01 messages
  2. An HTTP request to https://nsite.run/upload or https://nsite.run/{sha256} is handled by the blossom server
  3. A request to https://nsite.run (root domain, no npub subdomain) returns the SPA or a stub 200 confirming SPA routing
  4. A request to https://npub1xxx.nsite.run returns a stub 503 confirming npub subdomain routing is wired (nsite resolver not yet live)
  5. A request to https://identifier.npub1xxx.nsite.run is identified as a named-site request and routed to the resolver stub
**Plans**: 1 plan
Plans:
  - [ ] 04-01-PLAN.md — Hostname parser, stub handlers, router dispatch, main.ts wiring, build verification

### Phase 5: nsite Resolver and Progressive Caching
**Goal**: Visiting any npub or named-site subdomain on nsite.run resolves and serves the correct nsite content, with intelligent caching behavior and non-intrusive update notifications
**Depends on**: Phase 4
**Requirements**: GATE-01, GATE-02, GATE-03, GATE-04, GATE-05, GATE-06, GATE-07, GATE-08, GATE-09, GATE-10, GATE-11, GATE-12, CACHE-01, CACHE-02, CACHE-03, CACHE-04, CACHE-05, CACHE-06, CACHE-07, CACHE-08, CACHE-09
**Success Criteria** (what must be TRUE):
  1. Visiting an npub subdomain cold (never cached) shows a loading page with the user's profile name and avatar while resolving, then serves the site
  2. Visiting an npub subdomain that was previously cached but has an update serves the cached version immediately and injects a visible "click to refresh" banner into the HTML
  3. Visiting an npub subdomain with a current cached version serves immediately with no banner
  4. Visiting a named-site subdomain (identifier.npub1xxx.nsite.run) resolves and serves the correct named nsite
  5. Fetched manifests and blobs are persisted into the gateway's own relay and blossom so other gateways can retrieve them
**Plans**: 4 plans
Plans:
  - [ ] 05-01-PLAN.md — Utility modules: content-type detection, security headers, HTML pages/templates, banner injection
  - [ ] 05-02-PLAN.md — Data access: gateway DB layer (copied from relay), NIP-01 WebSocket client, npub decoder
  - [ ] 05-03-PLAN.md — Core resolver: cache state machine, streaming resolution pipeline, file serving, persistence
  - [ ] 05-04-PLAN.md — Integration: router wiring, build verification, human checkpoint

### Phase 6: SPA Deploy Interface
**Goal**: A user can authenticate with their Nostr identity and deploy a static site to nsite.run directly from their browser
**Depends on**: Phase 2, Phase 3
**Requirements**: SPA-01, SPA-02, SPA-03, SPA-04, SPA-05, SPA-06, SPA-07, SPA-08, SPA-09, SPA-10, SPA-11, SPA-12, SPA-13
**Success Criteria** (what must be TRUE):
  1. A user can authenticate with a NIP-07 browser extension or a NIP-46 bunker URL and deploy a folder or zip/tar.gz archive
  2. Before uploading, the SPA shows the full file list and rejects or warns when dangerous filenames or secret-like content patterns are detected
  3. After a successful deploy, the site is accessible at the user's npub subdomain on nsite.run
  4. An unauthenticated visitor to nsite.run sees educational content explaining what nsites are and links to other gateways, tools, and blossom servers
**Plans**: 5 plans
Plans:
  - [ ] 06-01-PLAN.md — SPA scaffold: Vite + Svelte 4 + Tailwind CSS project, Vitest test runner
  - [ ] 06-02-PLAN.md — File processing libs: SHA-256 hashing, ZIP/TAR.GZ extraction, secret scanning
  - [ ] 06-03-PLAN.md — Auth + deploy libs: Nostr signers (anonymous/NIP-07/NIP-46), BUD-02 upload, manifest publishing
  - [ ] 06-04-PLAN.md — Svelte UI components: deploy flow, auth modals, file tree, educational content
  - [ ] 06-05-PLAN.md — Gateway integration: serve SPA from Bunny Storage, CI/CD, human verification
