Create the gateway's database access layer and NIP-01 WebSocket client for external relay communication. These are the two data access channels the resolver uses: direct DB for the own relay, WebSocket for external relays.

Purpose: The gateway needs two distinct relay access modes. Direct DB queries (zero-latency, shared Bunny DB) for cache lookups and persistence. Raw NIP-01 WebSocket connections for querying seed relays and user outbox relays. Both are consumed by the resolver in Plan 03.

Output: Two data access modules: db.ts (copied from relay, adapted for gateway) and nostr-ws.ts (lightweight NIP-01 client + npub decoder).

    Import pattern: `import { nip19 } from "@nostr/tools/pure";`

    2. **`queryRelayOnce(relayUrl: string, filter: NostrFilter, timeoutMs: number): Promise<NostrEvent[]>`** — Raw NIP-01 WebSocket client per RESEARCH.md Pattern 4:
       - Generate random subscription ID: `crypto.randomUUID().slice(0, 8)`
       - Open `new WebSocket(relayUrl)`
       - On open: send `["REQ", subId, filter]`
       - On message: parse JSON. If `["EVENT", subId, event]` push event. If `["EOSE", subId]` close and resolve.
       - Set timeout timer: on timeout, close socket and resolve with collected events so far.
       - On error/close: clear timeout, resolve with collected events.
       - NEVER reject — always resolve with whatever events collected (empty array on total failure).

    3. **`queryMultipleRelays(relayUrls: string[], filter: NostrFilter, timeoutMs: number): Promise<NostrEvent[]>`** — Fan out `queryRelayOnce` to all URLs in parallel via `Promise.allSettled`. Collect all events, deduplicate by event.id using a Set. Return deduplicated array sorted by created_at DESC.
