<task type="auto">
  <name>Task 1: Create CI workflow with validation, build, size check, and PR delta comment</name>
  <files>.github/workflows/ci.yml</files>
  <action>
    Create `.github/workflows/ci.yml` with the following structure:

    **Trigger:** `on: push` (all branches) and `on: pull_request`

    **Job 1: `validate`** (runs on all triggers)
    - `runs-on: ubuntu-latest`
    - Steps:
      1. `actions/checkout@v4`
      2. `denoland/setup-deno@v2` with `deno-version: v2.x`
      3. `deno fmt --check`
      4. `deno lint`
      5. `deno check apps/relay/src/main.ts apps/blossom/src/main.ts apps/gateway/src/main.ts packages/shared/src/mod.ts`
      6. `deno test --allow-all --recursive`

    **Job 2: `build-and-size-check`** (runs on all triggers)
    - `runs-on: ubuntu-latest`
    - Steps:
      1. `actions/checkout@v4`
      2. `denoland/setup-deno@v2` with `deno-version: v2.x`
      3. `deno task build`
      4. `deno run --allow-read scripts/check-bundle-sizes.ts`
      5. Save bundle sizes to environment variables for PR comment:
         ```
         echo "RELAY_SIZE=$(stat -c%s apps/relay/dist/relay.bundle.js)" >> $GITHUB_ENV
         echo "BLOSSOM_SIZE=$(stat -c%s apps/blossom/dist/blossom.bundle.js)" >> $GITHUB_ENV
         echo "GATEWAY_SIZE=$(stat -c%s apps/gateway/dist/gateway.bundle.js)" >> $GITHUB_ENV
         ```

    **Job 3: `bundle-delta`** (runs ONLY on pull_request)
    - `runs-on: ubuntu-latest`
    - `if: github.event_name == 'pull_request'`
    - `permissions: pull-requests: write` (needed to post PR comment)
    - Steps:
      1. Checkout base branch: `actions/checkout@v4` with `ref: ${{ github.base_ref }}`
      2. `denoland/setup-deno@v2` with `deno-version: v2.x`
      3. Build base branch: `deno task build`
      4. Save base sizes to env vars:
         ```
         echo "RELAY_BASE=$(stat -c%s apps/relay/dist/relay.bundle.js 2>/dev/null || echo 0)" >> $GITHUB_ENV
         echo "BLOSSOM_BASE=$(stat -c%s apps/blossom/dist/blossom.bundle.js 2>/dev/null || echo 0)" >> $GITHUB_ENV
         echo "GATEWAY_BASE=$(stat -c%s apps/gateway/dist/gateway.bundle.js 2>/dev/null || echo 0)" >> $GITHUB_ENV
         ```
      5. Checkout PR branch: `actions/checkout@v4` (default, checks out PR merge commit)
      6. Build PR branch: `deno task build`
      7. Save PR sizes to env vars (RELAY_PR, BLOSSOM_PR, GATEWAY_PR)
      8. Post PR comment using `actions/github-script@v7`:
         - Find existing bot comment (update instead of creating duplicates) by searching for a marker like `<!-- bundle-size-report -->`
         - Generate markdown table with columns: Bundle | Base | PR | Delta | Status
         - Delta shows `+X.XKB` or `-X.XKB` with color indicators
         - Status column: checkmark for OK, warning for >750KB, X for >1MB
         - Use `github.rest.issues.createComment` or `github.rest.issues.updateComment`

    The `bundle-delta` job is informational — it should NOT block merging (no `needs:` from other jobs). The `build-and-size-check` job is what enforces the hard limit.
  </action>

<task type="auto">
  <name>Task 2: Create deploy workflow for Bunny Edge Script deployment</name>
  <files>.github/workflows/deploy.yml</files>
  <action>
    Create `.github/workflows/deploy.yml`:

    **Trigger:** `on: push: branches: [main]`

    **Job: `deploy`**
    - `runs-on: ubuntu-latest`
    - Steps:
      1. `actions/checkout@v4`
      2. `denoland/setup-deno@v2` with `deno-version: v2.x`
      3. `deno task build`
      4. `deno run --allow-read scripts/check-bundle-sizes.ts` (safety check before deploy — redundant with CI but catches edge cases)
      5. Deploy relay:
         ```yaml
         - name: Deploy relay
           uses: BunnyWay/actions/deploy-script@main
           with:
             script_id: ${{ secrets.BUNNY_RELAY_SCRIPT_ID }}
             deploy_key: ${{ secrets.BUNNY_RELAY_DEPLOY_KEY }}
             file: apps/relay/dist/relay.bundle.js
         ```
      6. Deploy blossom (same pattern with BLOSSOM secrets)
      7. Deploy gateway (same pattern with GATEWAY secrets)

    NOTE: Using `BunnyWay/actions/deploy-script@main` per research recommendation. Once initial setup is verified working, this should be pinned to a specific commit SHA for stability (tracked as a follow-up, not blocking).

    The deploy job does NOT depend on the CI workflow running first — GitHub branch protection rules should be configured to require CI to pass before merge to main. This is a user configuration step, not something the workflow controls.
  </action>
